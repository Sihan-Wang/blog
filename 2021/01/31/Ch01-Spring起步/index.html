Chapter 1: Spring起步

1.1 什么是Spring

Spring的核心是提供了一个容器container，称为Spring应用上下文Spring application context，会创建和管理应用组件（bean），bean装配起来形成完整的应用程序。这种装配通过基于依赖注入(dependency injection)的模式。
使用依赖注入的应用依赖于单独的实体（容器）来创建和维护所有的组件，并将其注入到需要他们的bean中。

  • @Configuration: 告诉Spring这是一个配置类，会为Spring应用上下文提供bean
  • @Bean: 标注配置类的方法，表明返回的对象以bean的形式添加到Spring应用上下文中。（一般bean的ID与方法名一致）

组件扫描（component scanning）：Spring能够自动发现Application类路径下的组件，将他们创建成Spring应用上下文中的bean。
自动装配（autowiring）：自动为组件注入他们依赖的其他bean
SpringBoot中的自动配置：基于类路径中各种因素合理猜测所需要配置的组件并装配到一起。

1.2 初始化Spring应用

注意到pom.xml文件中的dependecy
Spring Boot starter依赖，特别之处在于他们不包含库代码，而是传递性拉取其他库：

  • 构建文件会显著减小且易于管理，因为不必为每一个依赖库都声明依赖
  • 可以根据提供的功能思考依赖，而不是根据库的名称。
  • 不必担心库版本的问题，传递性引入的库的班恩是兼容的。
    plugin中，springboot插件：
  • 提供一个Maven goal，允许使用Maven来运行应用。
  • 确保依赖的所有库都包含在可执行JAR文件中，且保证在运行时类路径下是可用的。
  • 会在JAR文件中生成一个manifest文件，将引导类（TacoCloudApplication）声明为可执行JAR的主类

TacoCloudApplication.java:

@SpringBootApplication是下面三个注解的组合注解

  • @SpringBootConfiguration: 声明类为配置类
  • @EnableAutoConfiguration: 启用SpringBoot的自动配置。会告诉SpringBoot自动配置它认为会用到的组件
  • @ComponentScan: 启用组件扫描。可以用@Component、@Controller、@Service这样的注解声明其他类，Spring就会自动发现他们并注册为Spring应用上下文中的组件

main方法：是JAR文件执行时需要运行的方法。调用静态的run方法，执行应用引导过程，创建Spring应用上下文。（传给run的配置类不一定要和引导类相同）

测试应用：
https://docs.spring.io/spring-boot/docs/2.3.6.RELEASE/reference/html/spring-boot-features.html#boot-features-testing
SpringBoot的测试支持由两个模块提供：spring-boot-test包含核心项，spring-boot-test-autoconfigure支持测试的自动配置。
大多数程序员会使用spring-boot-starter-test作为“Starter”，它也会引入“vintage engine”从而可以支持JUnit4 & 5，如果已经将测试迁移到JUnit 5，就需要移除JUnit 4的支持：

这个测试类会执行必要的检查，确保Spring应用上下文能够成功加载。
@SpringBootTest，在需要SpringBoot特性时，可以替代标准Spring测试注释@ContextConfiguration。（@SpringBootTest会高速JUnit在启动测试的时候添加上SpringBoot的功能）
如果使用JUnit 4，需要加上@RunWith(SpringRunner.class)，否则这个注释会被忽略。@RunWith是JUnit的注解，会提供一个测试运行器（runner）指导JUnit进行测试。SpringRunner是Spring提供的测试运行器，会创建测试运行所需的Spring应用上下文。
如果使用JUnit 5，@SpringBootTest和其他@xxxTest已经包含了@ExtendWith(SpringExtension.class)，就不需要写这个注释了。

开始编写Spring应用

1、处理Web请求

Spring自带强大的Web框架Spring MVC，其核心是控制器(controller)的概念 – 处理请求并以某种方式进行信息响应的类。在面向浏览器的应用中，会填充可选的数据模型并将请求传递给一个视图，以便生成返回给浏览器的HTML。

@Controller: 让组件扫描将这个类识别为一个组件，创建一个HelloController实例作为Spring应用上下文的bean。（使用@Component、@Service、@Repository都可以起到类似的目的，作用是相同的，但是@Controller更能描述组件在应用中的角色）
@GetMapping(“/“): 如果针对”/“发送HTTP GET请求，这个方法会处理请求。
“home”会被解析为视图的逻辑名，这里可以使用Thymeleaf定义模板。最终形成的模板路径是：”/templates/home.html”

静态资源放到/src/main/resources/static文件夹下，所以图片的路径是/src/main/resources/static/images/1.png

3、测试控制器

@WebMvcTest注解是Spring Boot提供的特殊测试注解，会让这个测试在Spring MVC应用的上下文中执行：会将HomeController注册到Spring MVC中，就可以向它发送请求了。通过仿造出Spring MVC的运行机制，测试类被注入MockMvc，能够让测试实现mockup

http://localhost:8080

5、了解Spring Boot DevTools

  • 代码变更后会重启：DevTools运行时，应用程序会被加载到JVM两个独立的类加载器中，其中一个类加载器会加载你的Java代码、属性文件以及项目中/src/main/路径下几乎所有内容。这些条目很可能经常发生变化。另一个类加载器会加载依赖的库，这些库不太可能发生变化。探测到变更的时候，DevTools只会重新加载包含项目代码的类加载器，并重启Spring的应用上下文，所以可以减少应用启动的时间。但是依赖发生改变的时候还是需要重启应用。
  • 当面向浏览器的资源（如模板、JavaScript、样式表）等发生变化时，会自动刷新浏览器：DevTools在运行时会启动一个LiveReload服务，当与LiveReload浏览器插件结合的时候，就能在模板、图片、样式表、JavaScript等发生变化的时候自动刷新浏览器。
  • 自动禁用模板缓存：在配置时会缓存模板解析的结果，在为每个请求提供服务的时候就不需要重新解析模板，这在生产环境中是很好的方式。但是在开发过程中就会无法看到模板变更结果。
  • 如果使用H2数据库，内置了H2控制台：如果使用H2数据库开发，DevTools就会自动启动H2，访问ttp://localhost:8080/h2-console，就能看到应用所使用的数据。

小结

在pom.xml文件中，声明了对Web和Thymeleaf starter的依赖，会传递引入其他依赖（包括Spring MVC框架，嵌入式的Tomcat，Thymeleaf和Thymeleaf布局方言），还会引入Spring Boot的自动配置库，完成这些功能：

  • 在Spring应用上下文中配置bean以启动Spring MVC
  • 在Spring应用上下文中配置嵌入式Tomcat服务器
  • 配置Thymeleaf视图解析器，以便于用Thymeleaf模板渲染Spring MVC视图。

Spring家族

1、Spring核心框架

核心容器、依赖注入框架；Spring MVC，即Spring的Web框架；对数据持久化的基础支持，尤其是基于模板的JDBC支持；对反应式风格编程的支持，包括名为Spring WebFlux的反应式Web框架。

2、Spring Boot

starter依赖、自动配置

  • Actuator能够洞察应用运行时的内部工作状况，包括指标、线程dump信息、应用的健康状况、应用可用的环境属性
  • 灵活的环境属性规范
  • 在核心框架的测试辅助功能之上提供了对测试的额外支持。
  • 提供一个基于Groovy脚本的编程模型Spring Boot CLI(命令行接口)，可以将整个应用程序编写成Groovy脚本的集合，通过命令行去运行它们。

3、Spring Data

将应用程序的数据repository定义为简单的Java接口，在定义驱动存储和检索数据的方法时使用一种命名规定即可。
能够处理多种不同类型的数据库，关系型数据库JPA，文档数据库Mongo，图数据库Neo4j

4、Spring Security

解决了应用程序的通用的安全需求，包括身份验证、授权、API安全性。

5、Spring Integration和Spring Batch

为基于Spring的应用程序一些集成模式的实现，解决应用程序与其他应用甚至本应用中的其他组件进行集成的问题。
Spring Integration解决了实时集成的问题（数据在可用时就得到处理）。
Spring Batch解决的是批处理集成的问题（数据可以收集一段时间，直到某个触发器发出信号）

6、Spring Cloud

不将应用程序作为单个部署单元来开发，而是使用微服务组成的多个独立部署单元来组合形成应用程序。
